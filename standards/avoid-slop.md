# Gaudi AI Avoid Slop Standards

Use this file whenever designing, editing, or reviewing the Gaudi AI website.

The goal is to prevent the site from looking like a generic AI startup website. Gaudi should feel premium, modern, construction-native, operational, and trustworthy.

## Design read

Before making design changes, read the brief and state one sentence:

"Reading this as: B2B SaaS landing page for construction operators, with a premium, modern, construction-native language."

## Core direction

This is a marketing landing page for Gaudi AI.

The website should feel:

- Premium
- Modern
- Construction-native
- Operational
- Trustworthy
- Minimal but not empty
- Specific to construction workflows

The website should not feel:

- Generic AI SaaS
- Chatbot-first
- Sci-fi
- Cartoonish
- Overly animated
- Template-like
- Like a generic Vercel/v0 landing page

## Avoid generic AI design

Do not use:

- Purple AI gradients
- Neon glows
- Random mesh backgrounds
- Generic AI sparkles
- Chatbot-only hero visuals
- Three identical generic feature cards
- Fake dashboards made from meaningless boxes
- Stock construction photos as the main hero visual
- Cartoon hard hats, cranes, or builders
- Fake customer logos
- Made-up exact metrics unless clearly labeled as estimates

## Brand specificity

Every section should feel specific to Gaudi and construction.

Use concrete construction language:

- plans
- specs
- bids
- estimates
- proposals
- RFIs
- submittals
- draw inspections
- punch lists
- project updates
- subcontractor coordination
- document review
- bid leveling
- project admin
- field operations
- back-office workflows

Avoid vague startup language:

- revolutionary
- seamless
- unleash
- next-gen
- supercharge
- unlock the power of AI
- transform your business
- AI-powered everything
- effortless workflows

## Layout rules

Prefer:

- Asymmetric split hero
- Clear product workflow visual
- Document cards
- Workflow diagrams
- Estimate tables
- Bid leveling examples
- Construction command-center style visuals
- Muted backed-by section
- Clear 3-part how-it-works section
- Quantified benefits
- Interactive ROI calculator or workflow diagnostic

Avoid:

- Centered generic hero with vague tagline
- Repeating the same layout for every section
- Too many equal 3-column feature-card sections
- Overloading the hero with badges, stats, or small labels
- Putting trust logos inside the hero
- Massive empty sections with no useful message
- Decoration that does not explain the product

## Hero rules

The hero must communicate the product quickly.

Hero should include:

- A clear headline
- A short subheadline
- One primary CTA
- One secondary CTA at most
- A strong product or workflow visual

Hero should not include:

- Long paragraphs
- Too many badges
- Trust logo wall
- Feature list
- Fake version labels
- Decorative scroll cues
- Chatbot-only product mockup

The CTA should be visible without scrolling.

## Product visual rules

If showing product visuals, make them look like construction workflow tools.

Good product visual ideas:

- Plans and specs flowing into Gaudi
- Subcontractor bids becoming a bid leveling table
- Scope gaps being flagged
- Project documents becoming proposal drafts
- Estimate line items being structured
- Project admin tasks being generated
- Construction documents becoming a project context brain

Avoid:

- Generic chatbot box
- Meaningless fake dashboard rectangles
- Random AI orb
- Abstract sci-fi interface
- Fake UI that does not relate to construction

## Color rules

Use #c35a25 as the main Gaudi accent for logo and highlights.

Rules:

- Use one accent color consistently.
- Do not introduce random accent colors in different sections.
- Keep the palette restrained.
- Use strong contrast for text and buttons.
- Avoid generic purple/blue AI glow styling.
- Avoid pure black if a softer charcoal works better.

## Typography rules

Use clean, modern sans-serif typography.

Prefer:

- DM Sans (primary)
- Canva Sans (secondary, smaller copy)
- Times New Roman italic, for emphasis words only

Avoid:

- Defaulting to Inter unless already used in the project
- Random serif words inside sans-serif headlines beyond intentional emphasis moments
- Oversized headlines that wrap awkwardly
- Too many tiny uppercase labels

## CTA rules

Keep CTA labels consistent.

Primary CTA:

- Contact Us

Secondary CTA:

- Join Waitlist

Avoid using many labels for the same action, such as:

- Contact Us
- Get in Touch
- Let's Talk
- Reach Out
- Start Now

Pick one label per action and use it consistently.

Make sure CTA text does not wrap on desktop.

## Copy rules

Keep copy concrete, short, and construction-specific.

Good copy should answer:

- What does Gaudi do?
- Who is it for?
- What workflows does it automate?
- What input does it use?
- What output does the team get?
- Why does this save time or reduce admin drag?

Avoid:

- Hype without specifics
- Overpromising
- Fake precision
- Long paragraphs
- Generic SaaS copy
- Cute but unclear phrases

If using numbers, label them as estimates unless they are real.

Examples:

Good:
"Estimate potential admin time saved across estimating, bid leveling, and proposal workflows."

Bad:
"Save 92% of your time instantly."

## Benefit rules

Benefits should be specific and, where possible, quantified.

Good benefit examples:

- Reduce manual bid comparison time
- Shorten proposal drafting cycles
- Flag missing scope before award
- Turn plans and specs into structured project context
- Generate project updates from scattered documents

Avoid vague benefits like:

- Work smarter
- Move faster
- Unlock efficiency
- Scale with AI
- Transform operations

## Interactive module rules

The interactive module should be useful, not gimmicky.

Preferred modules:

- ROI calculator
- Construction admin drag calculator
- Workflow diagnostic
- Build your AI back office blueprint
- What should Gaudi automate for you quiz

The output should show:

- Recommended workflow
- Estimated admin hours
- Potential time savings range
- Suggested Gaudi automation path
- CTA to Contact Us or Join Waitlist

Use careful language:

- estimated
- potential
- directional
- based on your inputs

Do not overpromise exact savings.

## Motion rules

Use motion only when it helps explain:

- hierarchy
- storytelling
- feedback
- state change

Avoid:

- Motion just because it looks cool
- Excessive animations
- Infinite loops everywhere
- Scroll effects that make the site hard to use
- Animations that break on mobile

Respect reduced-motion preferences if adding motion.

## Accessibility rules

The site must be usable and readable.

Rules:

- Make the site mobile responsive.
- Ensure buttons have readable contrast.
- Ensure forms have labels above inputs.
- Do not use placeholder text as the only label.
- Keep navigation on one line on desktop.
- Make mobile layouts collapse cleanly.
- Do not make text too small.
- Do not rely on color alone to communicate meaning.

## Testimonial rules

If testimonials are included:

- Use real testimonials when available.
- If placeholders are used, clearly treat them as placeholders.
- Keep quotes short.
- Avoid generic fake names.
- Avoid fake company logos.

## Backed-by rules

If including a backed-by section:

- Keep it small and muted.
- Do not invent backers.
- Use real names or leave placeholders clearly labeled.
- The section should add credibility without taking over the page.

## Footer rules

The footer should feel intentional.

Include:

- Gaudi AI name or wordmark
- LinkedIn link
- Email or Contact Us link
- Privacy link if available

Avoid:

- Fake version numbers
- Decorative metadata
- Random weather, city, or timestamp strips
- Overly complicated footer layouts

## Final pre-flight check

Before finishing any page or component, check:

- Does this feel specific to construction?
- Does this avoid generic AI startup styling?
- Is the CTA clear?
- Is the page easy to skim?
- Are there any fake metrics?
- Are there any fake logos?
- Is the layout responsive?
- Are there too many animations?
- Is the hero clear within the first screen?
- Are we using #c35a25 consistently?
- Is the code organized into components?
- Does the product visual show real construction workflows?
- Would a GC, estimator, PM, or construction executive understand the value quickly?
